# Discovery Log 공통 작성 규칙

이번 갈루아 글을 Discovery Log의 첫 기준점으로 삼는다.

WindowMirror 본편이 독자의 경험을 만드는 글이라면, Discovery Log는 제작 과정에서 제작자의 시선이 달라진 순간을 남기는 기록이다.

개발 내역이나 완성도 평가를 정리하는 후기와 구분되어야 한다.

## 작성 시작 전: 적용 범위와 구조

Discovery Log는 과학정류장 카테고리 전체에 자동 적용하는 레이아웃이 아니다. 제작 과정, 재검토, 예상 밖의 장면, 제작자의 시선이 달라진 발견을 기록할 때만 사용한다. 설명형 글, 별도 시각화 페이지, 일반 안내 글은 목적에 맞는 다른 레이아웃을 선택한다.

Discovery Log로 판단했다면 Front Matter에 반드시 다음을 사용한다.

```yaml
layout: discovery-log
categories:
  - sciencestop
tags: [Discovery Log, 과학정류장]
```

`layout: single`은 Discovery Log에 사용하지 않는다. `discovery-log` 레이아웃은 제목의 외곽 기준선과 안쪽 읽기 폭을 분리하고, 본문 이미지가 넓게 전환될 수 있도록 한다.

대표 장면 이미지는 일반 Markdown 이미지 문법 대신 아래 구조를 쓴다. `src`에는 저장소의 최종 공개 경로를 사용하고, 캡션이 필요할 때만 `figcaption`을 추가한다.

```html
<figure class="discovery-log__wide-image">
  <img src="/assets/images/sciencestop/example.png" alt="핵심 장면을 설명하는 짧은 대체 텍스트">
  <figcaption>필요할 때만 넣는 짧은 캡션</figcaption>
</figure>
```

이 구조는 본문 읽기 폭보다 넓게 이미지를 배치하고, 이미지 위아래 여백을 공통 CSS로 유지한다. 복사 시작점은 [Discovery Log 템플릿](../../docs/templates/discovery-log-template.md)이다.

## 1. 첫 문장은 반드시 장면으로 시작한다

작업 소개나 목적 설명으로 시작하지 않는다.

당시의 화면, 손의 움직임, 떠오른 단어, 예상과 다른 결과처럼 실제 순간이 먼저 보여야 한다.

좋은 예:

```text
계수를 하나 바꾸자
그래프와 근이 함께 움직였다.
```

피해야 할 예:

```text
이번 글에서는 첫 Experience Bridge의 제작 과정을 돌아본다.
```

## 2. 글 전체에는 발견이 하나만 있어야 한다

한 글 안에 여러 교훈이나 결론을 병렬로 나열하지 않는다.

버튼, 설명, 이미지, 팟캐스트처럼 여러 사례가 등장할 수는 있지만, 모두 하나의 발견을 향해 모여야 한다.

갈루아 글의 중심 발견:

```text
설명이
경험보다 먼저 와 있었다.
```

## 3. 모든 사례는 발견을 증명하는 장면으로 사용한다

사례 자체를 자세히 설명하는 것이 목적이 아니다.

사례는 제작자의 사고가 어떻게 바뀌었는지를 보여주는 장면이어야 한다.

다음과 같은 항목식 후기 구조는 피한다.

* 잘된 점
* 아쉬운 점
* 개선할 점
* 다음 계획

대신 다음 흐름을 따른다.

```text
당시 장면
→ 그때의 생각
→ 다시 보니 드러난 공통점
→ 하나의 발견
```

## 4. 평가보다 다시 보며 발견한 것을 쓴다

다음과 같은 평가 문장을 줄인다.

```text
잘 만들었다.
부족했다.
성공적이었다.
아쉬웠다.
```

대신 다음처럼 바꾼다.

```text
그때는 충분하다고 생각했다.
지금 다시 보니 다른 것이 먼저 보였다.
처음에는 서로 다른 문제라고 생각했다.
그런데 모두 같은 곳을 가리키고 있었다.
```

## 5. 설명보다 당시의 공기를 남긴다

Discovery Log는 기능 명세나 기술 문서가 아니다.

구조, 단계, 도구 사용법을 모두 설명하지 않는다.
본편에서 이미 확인할 수 있는 내용은 반복하지 않는다.

설명은 발견을 이해하는 데 필요한 만큼만 남긴다.

특히 과학·수학 설명은 지식을 가르치는 데까지 확장하지 않는다.
제작자가 어떤 연결을 놓쳤는지가 보이는 정도로만 사용한다.

## 6. 도구보다 발견을 먼저 쓴다

NotebookLM, 이미지 생성 도구, 코드, 프레임워크 이름이 글의 주인공이 되지 않도록 한다.

도구 이름은 장면과 발견이 먼저 나온 뒤 필요한 위치에서만 언급한다.

좋은 흐름:

```text
오디오 팟캐스트는
다른 방식으로
같은 사실을 보여주고 있었다.

갈루아 편을 만들며
NotebookLM의 대화는...
```

피해야 할 흐름:

```text
이번에는 NotebookLM을 사용했다.
NotebookLM의 기능은...
```

## 7. 문단은 문장 수가 아니라 호흡으로 나눈다

모든 문장 뒤에 빈 줄을 넣지 않는다.

의미가 이어지는 문장은 한 문단으로 묶고, 발견이나 장면이 전환되는 곳에서만 여백을 둔다.

핵심 발견 문장은 앞뒤에 충분한 빈 줄을 두어 시각적으로도 기억에 남도록 한다.

예:

```text
Experience Bridge를 다시 움직여 보며
하나의 공통된 문제가 보였다.


설명이
경험보다 먼저 와 있었다.


오디오 팟캐스트는...
```

## 8. 결론을 교훈으로 닫지 않는다

독자에게 무엇을 해야 한다고 가르치거나, 제작 경험을 보편적인 성공 법칙으로 확장하지 않는다.

발견 이후 무엇을 다르게 보게 되었는지, 다음 항해가 어디에서 시작되었는지만 남긴다.

좋은 예:

```text
처음 만든 브릿지를
다시 움직여 보니

다음 항해는
이미 그 안에 있었다.
```

## 9. ‘오늘의 발견’은 항상 한 문장이다

한 개의 발견만 남긴다.

설명, 해설, 부연 문장을 추가하지 않는다.

두 줄로 끊을 수는 있지만 문장은 하나여야 한다.

예:

```text
설명을 덜어냈을 때
비로소 경험이 움직이기 시작했다.
```

Discovery Log가 여러 편 쌓였을 때 ‘오늘의 발견’만 읽어도 제작자의 시선이 어떻게 이동했는지 이어져야 한다.

## 10. 마지막 링크는 설명하지 않고 다음 항해만 가리킨다

원문 내용을 다시 설명하거나 클릭을 권유하는 문장을 길게 쓰지 않는다.

Discovery Log의 여운을 유지한 채 다음 글로 이어지게 한다.

예:

```text
갈루아 정류장으로 이어집니다.
```

피해야 할 예:

```text
이 회고의 바탕이 된 원문에서 Experience Bridge와 오디오 팟캐스트를 자세히 확인할 수 있습니다.
```

## 11. Discovery Log 최종 점검 질문

게시 전에 다음을 확인한다.

* 첫 문장이 실제 장면인가
* 글 전체의 발견이 하나인가
* 모든 사례가 같은 발견으로 모이는가
* 개발 후기나 기능 설명처럼 읽히는 부분은 없는가
* “잘했다/아쉬웠다”보다 “다시 보니 보였다”가 중심인가
* 도구 이름이 발견보다 먼저 나오지 않는가
* 핵심 발견 앞뒤에 충분한 호흡이 있는가
* 마지막이 교훈이나 성과 평가로 닫히지 않는가
* ‘오늘의 발견’이 정확히 한 문장인가
* 마지막 링크가 다음 항해만 가리키는가
* `layout: discovery-log`를 사용했는가
* 대표 장면 이미지가 필요하다면 `discovery-log__wide-image` figure 구조를 사용했는가
* `npm run validate:discovery-logs`에서 경고가 없는가
